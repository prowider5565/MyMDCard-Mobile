# Project Development Rules

## Decision Making

- Always ask for system/design-related decisions. Never assume or decide on behalf of the user.
- If anything is ambiguous, even something small, ask for clarification before proceeding.

## Code Quality

- Never write duplicate code. If an existing function or pattern already handles what is being requested, notify the user instead of reimplementing.
- Folder structure must be clean, logical, and easy to navigate.
- Code must be readable, simple, and easy to follow.

## Technology Choices

- When multiple packages, tools, or approaches exist for a given implementation, present all viable options with their advantages and trade-offs.
- Wait for the user's decision before proceeding with any option.

Always read the rule before implementing the request and say "Proceeding with request with the given project development rule"
